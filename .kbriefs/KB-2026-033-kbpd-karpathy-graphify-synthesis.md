# KBPD + Karpathy Wiki + Graphify: A Coherent Three-Layer Stack

## Purpose

Establish that KBPD, the Karpathy LLM Wiki method, and Graphify are non-redundant and
complementary. Record the full stack, the mapping, and the two gaps worth closing.

---

## The Three Layers

Each technique answers a different question:

| Layer | Technique | Question answered |
|---|---|---|
| Decision knowledge | KBPD (`.kbriefs/`) | When to create knowledge; what questions to ask before building |
| Knowledge maintenance | Karpathy wiki conventions | How to maintain synthesised knowledge (LLM-owned, lint, immutable sources) |
| Structural code knowledge | Graphify / god nodes | What the codebase structurally is (auto-extracted, per session) |

They are non-redundant. KBPD won't tell an agent where `MTARPSession` sits in the call
graph. Graphify won't tell it why MTARP was repositioned as an A2A extension. The two
layers are genuinely complementary.

---

## KBPD Already Is the Karpathy Wiki

The `.kbriefs/` directory is the Karpathy `wiki/` layer. The mapping is direct:

| Karpathy concept | KBPD equivalent |
|---|---|
| `raw/` — immutable sources | Research articles and docs cited in KBs |
| `wiki/` — LLM-maintained synthesis | `.kbriefs/*.md` |
| `index.md` — content catalog | KB index in `MEMORY.md` + KB numbering system |
| `log.md` — append-only record | KB `updated:` field + git history |
| Schema / process definition | The KBPD methodology itself |
| Lint — contradictions, orphans | **Gap: currently informal** |

This is not a new thing to build. It is a recognition that the existing KBPD practice
already implements the Karpathy pattern. Two conventions are missing (see below).

## The Full Stack for aider-relay

```
KBPD (.kbriefs/)         — project decisions, evidence, gap tracking
Graphify (GRAPH_REPORT)  — structural code orientation per session
TASK.md                  — per-task spec combining both layers
```

A TASK.md in this stack has three sources of context:

1. **Relevant K-Briefs** — link the KBs whose decisions govern this task
2. **God nodes from GRAPH_REPORT.md** — 5-10 structural concepts the agent must
   understand first (auto-generated by Graphify or hand-written)
3. **Task-specific spec** — objective, acceptance criteria, workflow, constraints
   (per KB-2026-031 template)

## Two Gaps Worth Closing

### Gap 1: No lint step

The Karpathy wiki lint operation (find contradictions, orphaned pages, stale claims) has
no equivalent in the current KBPD workflow. K-Briefs can go stale as code evolves and
no process surfaces this.

**Fix:** Add a periodic KB lint pass — a prompt that reads all K-Briefs, flags
contradictions with current code, and identifies KBs with no inbound links from
TASK.md or other KBs. Run before starting a major implementation phase.

### Gap 2: No immutable / agent-maintained tagging

TASK.md files have no convention distinguishing sections the agent must not modify
(background context, constraints) from sections it can update during a session (log,
progress, notes).

**Fix:** Add `[IMMUTABLE]` and `[AGENT-MAINTAINED]` markers to TASK.md sections.
Mirrors the Karpathy `raw/` vs `wiki/` split.

---

## Why This Stack Is Unusual

Most teams have one of:
- Flat docs (README, CLAUDE.md)
- Vector RAG over a corpus

This stack has:
- A formal methodology for knowing *when* to write new knowledge (KBPD gap
  identification before building)
- A curated LLM-maintained KB with evidence provenance (Karpathy wiki / .kbriefs/)
- Auto-extracted structural orientation that doesn't go stale (Graphify)
- A per-task spec that combines all three into agent-consumable form (TASK.md)

The combination is not common in documented practice as of 2026-05-01.

## Updated TASK.md Template Additions (from KB-2026-031 base)

```markdown
## Orientation
<!-- God nodes: 5-10 structural concepts the agent must understand first -->
<!-- Source: GRAPH_REPORT.md or hand-written -->
- God nodes: [list]
- Key relationship: [one non-obvious dependency]
- Non-obvious: [one thing that would surprise a reader]

## Knowledge Briefs
<!-- KBs whose decisions govern this task -->
- KB-2026-0XX: [one-line relevance]

## Agent Log  <!-- [AGENT-MAINTAINED] -->
<!-- Append-only. Agent writes one line after each logical unit of work. -->
<!-- Format: [HH:MM] action | file or component touched -->
- [start] Beginning task
```

Sections without `[AGENT-MAINTAINED]` are implicitly `[IMMUTABLE]`.
