# Higher-Level Context Relay Design Space

## Context

Claude Code and Codex are agentic coding assistants. Each maintains its own view of the codebase within a session. When switching from one to the other due to usage exhaustion, we cannot simply transfer a messages array. The incoming provider needs enough context to continue the task coherently. This is the dominant unsolved design problem in aider-relay.

## Problem Statement

When Provider A (e.g. Claude Code) exhausts its usage window mid-task, Provider B (e.g. Codex) must:
1. Understand the overall goal
2. Know what has already been done
3. Know what remains to be done
4. Understand the current state of the files
5. Continue without requiring the user to re-explain

What is the minimum viable context that enables this, and how is it stored and injected?

## Design Space Dimensions

- **Completeness**: How much of the prior session's work is conveyed to the new provider
- **Reliability**: Does the new provider actually use the context correctly
- **Storage complexity**: How hard is it to maintain the context store
- **Latency on switch**: How long does handoff take
- **Lossiness**: What information is unavoidably lost

## Context Categories

### Category 1: Git State (Objective, Lossless)
The git repository captures all file changes made so far. The new provider can `git diff`, `git log`, `git status` to understand what was changed.

- **Availability**: Always present in a git repo
- **Completeness**: Complete for file changes, zero for intent/reasoning
- **Injection method**: Tell the new provider to run `git log --oneline -20` and `git diff HEAD~N`

### Category 2: Task Description (Intent, User-Provided)
The original user request / task goal. This is the most important context — without it, the new provider doesn't know what it's supposed to accomplish.

- **Availability**: Must be captured at task start
- **Completeness**: Complete if captured at start; may drift as task evolves
- **Injection method**: Prepend to the initial prompt sent to the new provider

### Category 3: Subtask/Progress State (Structured, Maintained)
A structured record of: what subtasks were planned, which are complete, which are in-progress, which remain. Analogous to a todo list.

- **Availability**: Must be actively maintained by the relay during the session
- **Completeness**: As complete as the relay's tracking
- **Injection method**: Serialise as a markdown checklist in the handoff prompt

### Category 4: Key Decisions and Constraints (Reasoning, Semi-Structured)
Important choices made during the session: "we decided to use approach X because Y", "we discovered that Z doesn't work because W". Without these, the new provider may re-explore dead ends.

- **Availability**: Must be extracted from session events (hard to automate reliably)
- **Completeness**: Lossy — hard to extract programmatically from agentic output
- **Injection method**: Summarised in handoff prompt; could be LLM-generated from session transcript

### Category 5: Session Transcript Summary (History, Lossy)
A condensed version of the prior session. Could be generated by asking the exhausted provider to summarise before it fully stops, or by post-processing the session log.

- **Availability**: Session logs available from both CLIs
- **Completeness**: Lossy — summarisation loses detail
- **Injection method**: Include in handoff prompt

## Options in the Space

### Option 1: Git-Only Context (Minimal)

Rely entirely on git state. Handoff prompt:
```
Continue the following task in this repository:
<task description>

Recent git history shows what has been done:
<git log output>

Current uncommitted changes:
<git diff>

Please continue from where this left off.
```

- **Completeness**: Low — no reasoning, no subtask tracking
- **Reliability**: Medium — new provider may duplicate work or miss intent
- **Storage complexity**: Zero — git already exists
- **Latency on switch**: Fast — just run git commands

### Option 2: Structured Context File (RelayContext)

Maintain a `relay-context.json` file (or `.aider-relay/context.json`) that is updated throughout the session:

```json
{
  "task": "Add OAuth login to the API",
  "status": "in_progress",
  "started_at": "2026-04-27T14:30:00Z",
  "provider_history": [
    {"provider": "claude-code", "session_id": "abc123", "exhausted_at": "2026-04-27T16:45:00Z"}
  ],
  "subtasks": [
    {"id": 1, "description": "Add /auth/login endpoint", "status": "done"},
    {"id": 2, "description": "Add JWT token generation", "status": "in_progress"},
    {"id": 3, "description": "Add /auth/refresh endpoint", "status": "pending"}
  ],
  "key_decisions": [
    "Using PyJWT library for token generation",
    "Refresh tokens stored in Redis, not in DB"
  ],
  "files_in_scope": ["src/api/auth.py", "src/models/user.py", "tests/test_auth.py"]
}
```

Handoff prompt generated from this structure.

- **Completeness**: High if maintained correctly
- **Reliability**: High — structured data produces reliable prompts
- **Storage complexity**: Medium — relay must update context as session progresses
- **Latency on switch**: Fast — context file is ready

### Option 3: LLM-Generated Session Summary

Before (or just after) detecting exhaustion, ask the current provider (or a separate LLM call) to summarise the session:

```
Summarise what was accomplished in this session, what decisions were made, 
and what remains to be done. Format as a handoff brief.
```

Problem: if the provider is exhausted, it may refuse to process another request. Must be triggered proactively (e.g. every N turns) rather than reactively.

- **Completeness**: Medium — LLM summarisation is lossy but captures reasoning
- **Reliability**: Medium — depends on summarisation quality
- **Storage complexity**: Low — just store the summary string
- **Latency on switch**: High if triggered reactively (provider may be unavailable)

### Option 4: Combined (Option 2 + Option 3 Proactive)

Maintain Option 2 structure throughout, plus trigger a summary every N turns (e.g. every 10 turns) and store it in the context file. On switch, inject both the structured state and the latest summary.

- **Completeness**: Highest
- **Reliability**: Highest
- **Storage complexity**: Medium-High
- **Latency**: Fast on switch (summaries are pre-computed)

## Design Space Map

| Option | Completeness | Reliability | Storage | Switch Latency | Phase |
|--------|-------------|-------------|---------|----------------|-------|
| 1: Git-only | Low | Medium | Zero | Fast | 1 |
| 2: Context file | High | High | Medium | Fast | 2 |
| 3: LLM summary | Medium | Medium | Low | High | 3 |
| 4: Combined | Highest | Highest | Medium | Fast | 4 |

## Convergence Strategy

**Phase 1:** Git-only context. Validate that handoff produces usable results before investing in context tracking infrastructure.

**Phase 2:** Add `RelayContext` structured file. Populate task, subtasks, files_in_scope manually at task start; update subtask status during session.

**Phase 3:** Auto-populate key_decisions from session events. Add proactive periodic summarisation.

## RelayContext Schema (Phase 2 Target)

```python
@dataclass
class SubTask:
    id: int
    description: str
    status: str  # "pending" | "in_progress" | "done" | "blocked"

@dataclass
class ProviderRun:
    provider: str
    session_id: str
    started_at: str
    exhausted_at: str | None

@dataclass
class RelayContext:
    task: str
    status: str
    started_at: str
    provider_history: list[ProviderRun]
    subtasks: list[SubTask]
    key_decisions: list[str]
    files_in_scope: list[str]

    def to_handoff_prompt(self) -> str:
        """Generate a handoff prompt for the incoming provider."""
        done = [t for t in self.subtasks if t.status == "done"]
        pending = [t for t in self.subtasks if t.status != "done"]
        return f"""
You are continuing a task in this repository. A previous AI assistant was working on this and hit a usage limit.

## Task
{self.task}

## Already Completed
{chr(10).join(f"- [x] {t.description}" for t in done)}

## Remaining
{chr(10).join(f"- [ ] {t.description}" for t in pending)}

## Key Decisions Made
{chr(10).join(f"- {d}" for d in self.key_decisions)}

## Files in Scope
{chr(10).join(f"- {f}" for f in self.files_in_scope)}

Please continue from where the previous assistant left off. Run `git log --oneline -10` and `git diff HEAD~{len(done)}` to see what has already been done.
"""
```

## Critical Open Questions

1. Can we reliably extract subtask completion from agentic session events, or must it be manually tracked?
2. When Claude Code is exhausted, can we still issue one final "summarise" request, or is the session fully blocked?
3. Does the Codex SDK provide structured output (JSONL) that makes event parsing reliable enough for auto-tracking?

## Applicability

- ✅ The central design challenge for aider-relay
- ✅ Must be validated with Phase 1 (git-only) before committing to Phase 2
